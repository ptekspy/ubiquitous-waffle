# v0.3.0 persistence and planner queue

## Database

v0.3.0 uses Prisma 7 and PostgreSQL. Set `DATABASE_URL`, `BETTER_AUTH_SECRET`, and `BETTER_AUTH_URL` in `.env.local`, then run:

```bash
pnpm install
pnpm db:generate
pnpm db:migrate
pnpm dev
```

Saved data is split into:

- Better Auth `User`, `Session`, `Account`, and `Verification` tables
- `User.redditUsername` for the workspace default Reddit account
- `RedditAccount` mapped to the existing `Account` table
- `AccountScan`
- `PostSnapshot`
- `CommentSnapshot`
- `SubredditSnapshot`
- `MediaGroup`
- `PlannerJob`

The scanner stores the cleaned analytics rows, not the raw textarea value. New scans are scoped to the signed-in user.

## Auth

Better Auth handles email/password accounts at `/api/auth/*`.

Local verification uses the Email OTP plugin. On sign-up or resend, the verification code is printed to the terminal running `pnpm dev`.

Required auth env:

```bash
BETTER_AUTH_SECRET="replace-with-at-least-32-random-characters"
BETTER_AUTH_URL="http://localhost:3000"
```

## Workspace dashboard

The signed-in app mounts the authenticated dashboard only after the Better Auth session is available.

On mount it calls:

```http
GET /api/workspace
```

That route returns:

- the saved `redditUsername`
- the latest persisted `AnalyzeResponse`, rebuilt from PostgreSQL

The UI uses that response to prefill the username field and render the dashboard immediately after refresh.

When a scan succeeds, the server stores the scanned Reddit username on the signed-in `User`, so the user only needs to set it once.

## Planner queue

Scan imports create a `PlannerJob` with status `QUEUED`.

The website does not call Ollama directly during the scan request. A worker calls:

```http
POST /api/planner/jobs/process
Authorization: Bearer <PLANNER_WORKER_SECRET>
```

That endpoint claims one queued job, calls the configured Ollama service, stores the result, and exits.

Local development can omit `PLANNER_WORKER_SECRET`. Production should set it.

## Persistent worker

Run the app in one terminal:

```bash
pnpm dev
```

Run the planner worker in another terminal:

```bash
pnpm worker:planner
```

The worker keeps polling `/api/planner/jobs/process`, processes one queued job at a time, logs the result, then waits before checking again.

Optional worker env:

```bash
PLANNER_WORKER_TARGET="http://localhost:3000"
PLANNER_WORKER_INTERVAL_MS="5000"
PLANNER_WORKER_ERROR_INTERVAL_MS="15000"
```

## Ollama env

```bash
OLLAMA_BASE_URL="https://ollama.tik-track.com"
OLLAMA_PLANNER_MODEL=""
OLLAMA_API_KEY=""
PLANNER_WORKER_SECRET=""
```

If `OLLAMA_PLANNER_MODEL` is set, that exact model is used.

If `OLLAMA_PLANNER_MODEL` is blank, the processor asks `/api/tags` and chooses the best available planner model by scoring names and size. It prefers general instruction models such as Qwen, Dolphin, Nous/Hermes, Llama, Mistral, and Gemma, and avoids embedding, vision, and code-specialist models for this planning task.

## Job status

The dashboard receives the queued job id with the scan response and polls:

```http
GET /api/planner/jobs?jobId=<job-id>
```

When the worker stores a completed result, the dashboard card renders the strategy JSON.
